## フォルダ構成

開発上で特に重要なフォルダ・ファイルについて記載

- apps/       他のパッケージから参照されないパッケージを配置。配下のパッケージ名は `@apps/フォルダ名` で統一
  - web/     ：＜Web アプリの説明＞
- packages/ 他のパッケージから参照されるパッケージを配置。配下のパッケージ名は `@packages/フォルダ名` で統一
  - supabase ：Supabase の設定と supabase の薄いラッパー

## 開発フロー

開発時は必ず以下のフローを遵守すること

### 準備

- 不要な場合を除いて開発対象の開発サーバーを起動

### 開発

以下のチェックを順番に行い全てがパスするまで開発を続ける
なお、新しい機能の開発や変更時は必要な e2e 、ユニットテストの作成・修正まで必須

- /verify コマンドで開発した機能の動作確認
- lint、フォーマット等のバリデーションチェック（validate:check）
- ユニットテスト（test:unit:対象パッケージ）
- E2E テスト（test:e2e:clean:対象パッケージ）


### コード整理

- /simplify でコードを整理
- コードレビュー
  - Opus モデルのサブエージェント
  - `agy -p` の外部エージェント
- コードレビュー結果のレビュー
  - コードレビュー結果を再評価し妥当性のチェック
- コードレビューにより妥当な指摘があれば開発セクションに移動して修正
  - 最大３回まで
  - ３回目に妥当な指摘事項があれば後続のレポートに記載

### レポート

- ユーザーに以下の内容をまとめ結果報告
 - 作業概要
 - 開発中に行った重要な意思決定
 - 動作確認手順
 - 実際に動作確認を行った際のスクリーンショット
 - 今回の作業中にエージェントが失敗した事や時間がかかった事など改善点
- ローカルレビューの起動
  - `difit --keep-alive --include-untracked --background .`

## 調査依頼の回答、実装計画、結果の報告

ルールにしたがって HTML ファイル１枚として作成すること
また、作成後はユーザーにパス（file:///）を表示すること
なお、HTML ファイルで作成するのはユーザーが理解、判断をしやすくするためであり、 過剰な装飾求めるものではない

### ルール

- HTML ファイルの作成フォルダ： .work/reports/
- グラフ、テーブル、シンタックスハイライト、diff 表示、SVG などを活用してユーザーに理解しやすい表現を行うこと
  - グラフ:mermaid
  - 色付け:shiki
  - diff:diff2html（side-by-side）
  - コード提示する場合は行数も表示する
- セクションごとにカードブロックで分離する
- 横幅は制限せずウィンドウの横幅をフルで活用する
- コードの変更を記載するとき既存コードと変更予定コードを diff で表示する
