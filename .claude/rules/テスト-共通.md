# テスト： 共通

## テストでは実装のからのインポート避け、テスト内で定義する

悪い例

```ts
import { MAX_ITEM_COUNT } from "./constants";

expect(items).toHaveLength(MAX_ITEM_COUNT);`
```

良い例

```ts
// もしくはテスト内で MAX_ITEM_COUNT を定義
expect(items).toHaveLength(100);
```

**理由:**

1. **意図しない変更の検出**: 実装の定数が変わったときにテストが壊れることで、変更に気づける。インポートしていると自動追従してしまい、バグを見逃す
2. **テストは仕様のドキュメント**: 「100件表示される」は仕様であり、テストで明示すべき。

## 実装ではなく、ユーザーから見た振る舞い（外部から観察可能な動作）をテストする

- 悪い例: maxlength 属性が設定されている、className に foo-bar が設定されている
- 良い例: 最大文字数を超えて入力できない、VRT による検出

## グループ化（describe）は避け、test.extend を優先する

共通のセットアップ、ティアダウンを行いたい場合は test.extend を使ったフィクスチャーパターンを使用する

## モックを避ける

### 本物を使えるものは本物を使う

- DB アクセスはローカル DB（Testcontainers・ローカルの Postgres / SQLite など）を使う
- HTTP リクエストはローカルサーバー（テスト用に立てた実サーバー）を使う
- ファイル I/O はテンポラリディレクトリを使う
- 同一プロセス内の関数・モジュールは直接呼ぶ（依存注入のためだけの抽象化やモックは作らない）

### モックを許容するケース

- 外部 API（呼び出しに費用・時間・副作用が発生するもの）
- 時計・乱数・UUID 生成など、テストで決定性が必要な非決定的要素
- 本物を起動するコストが著しく高い、もしくはローカルで再現不可能な依存

これらの場合も、可能な限り **公式に提供されているフェイク実装**（例: Stripe の Test Mode、AWS の LocalStack、ライブラリ提供の in-memory 実装）を優先し、自前のモックは最後の手段とする。
