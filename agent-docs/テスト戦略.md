# テスト戦略

## 概要

自動テストの粒度・構造・実装スタイルを統一し、信頼性と保守性の高いテストスイートを構築するための戦略。

## 1. テストサイズの分類

### 方針

Google のテストサイズ ([Small / Medium / Large](https://testing.googleblog.com/2010/12/test-sizes.html)) の考え方を採用し、各テストがどのサイズに該当するかを意識して設計・実装する。

| サイズ | 実行範囲                           | 外部依存                                                                     | 実行時間の目安 |
| ------ | ---------------------------------- | ---------------------------------------------------------------------------- | -------------- |
| Small  | 単一プロセス・単一スレッド内で完結 | 不可（ネットワーク・ファイル I/O・DB・スリープ・マルチスレッドはすべて不可） | 数十ミリ秒以内 |
| Medium | 単一マシン内で完結                 | localhost に閉じたネットワーク、ファイル I/O、ローカル DB は可               | 数秒以内       |
| Large  | 複数マシン・外部システムを含む     | 外部 API、外部 DB、ブラウザ越しの E2E など制約なし                           | 数秒〜数分     |

## 2. テストサイズの優先順位

### 方針

Medium サイズのテストをベースとして優先的に実装する。Small は補助的に、Large は最小限に留める。

### 理由

- **Medium を主軸にする理由**: 実プロダクトに近い構成（ローカル DB やローカルサーバーを含む）で検証できるため、結合バグやスキーマ不整合などプロダクトの破壊につながりやすい不具合を高い確度で検知できる。Small だけでは過剰なモックに依存し、リファクタ耐性が低く偽陽性・偽陰性が増える
- **Small を補助にする理由**: 純粋ロジック（バリデーション、計算、変換）は Small で網羅した方が高速かつ安定する。ただし Small は「依存を全てモックした結果、本番経路を通っていない」テストになりやすいため、振る舞いの保証ではなくロジックの網羅に用途を限定する
- **Large を最小限にする理由**: 実行時間・安定性・メンテナンスコストすべてで不利なため、ユーザー操作シナリオなど Medium で代替不能な範囲に絞る。Large でしか守れない契約以外は Medium に押し下げる

## 3. テスト種別の概要

ファイル命名規則の詳細は [nextjs-命名規則.md](./nextjs-命名規則.md) を参照。

| 種別                  | ファイル名パターン                                                                          | ツール                    | 外部依存       |
| --------------------- | ------------------------------------------------------------------------------------------- | ------------------------- | -------------- |
| E2E テスト            | `*.e2e.test.ts`                                                                             | Playwright                | ブラウザ + DB  |
| small ブラウザテスト  | `*.client.test.ts`（hook・スクリプト）/ `*.[client\|universal].test.tsx`（コンポーネント） | Vitest（ブラウザモード）  | なし（モック） |
| medium サーバーテスト | `*.medium.server.test.ts`                                                                   | Vitest（node ランタイム） | DB あり        |
| small サーバーテスト  | `*.small.server.test.ts`                                                                    | Vitest（node ランタイム） | なし（モック） |

### 判断フロー

```
テストしたい内容は何か？
├─ 複数レイヤーにまたがる機能フロー（ページ遷移・DB 書き込み含む）
│    → E2E テスト
├─ クライアントコンポーネント / React Hook
│    → small ブラウザテスト
├─ サーバーロジック（DB への実際のアクセスが必要）
│    → medium サーバーテスト
└─ サーバーロジック（DB アクセスなし：権限・バリデーション・エラーハンドリング）
     → small サーバーテスト
```

## 4. カバレッジ

### 方針

- **E2E テスト以外のテスト**のカバレッジを測定する（E2E テストでは測定しない）
- E2E 以外のテストにおいて、**ブランチカバレッジ（Branch Coverage / C1）で 80% 以上**を維持する

### 理由

- **E2E を測定対象外とする理由**: E2E はシナリオ単位での価値検証が主目的であり、コード網羅率は適切な指標ではない。またフロントとサーバーで別プロセスにまたがるカバレッジの収集・統合は計測パイプラインの構築・維持コストが大きい
- **ブランチカバレッジを指標にする理由**: ステートメント / 行カバレッジでは三項演算子・`&&` / `||` の短絡評価・`?.` / `??` など式レベルの分岐を取りこぼすため
- **80% を下限とする理由**: 100% を目標にすると到達困難なエラーパスのカバーに労力が割かれ費用対効果が悪化する。一方 50〜60% 台では主要分岐の取りこぼしが常態化する

## E2E テスト

**対象：** ページ単位、ページ遷移単位

Layout・Page・Async Component・DB・Server Action など複数のレイヤーが結合された状態で検証するシナリオ。コンポーネント単体で検証できる内容は対象外。

対象

- 検索ページで条件を入力して一致する検索結果が表示される
- 登録フォームで入力して送信すると詳細ページに遷移する

対象外

- クライアントコンポーネントの登録フォームで必須入力が未入力で送信ボタンを押すとバリデーションエラーが表示される
  - コンポーネント単体で検証できるので small ブラウザテストで対応

## small ブラウザテスト

**対象：** クライアントコンポーネント、ユニバーサルコンポーネント、React Hook、ブラウザ用の関数

ネットワーク通信などが不要なテストケース、e2e で担保しにくいケース

対象

- テキストフィールドが 0 文字・最大文字数超えのときエラーが表示される
- 送信中はボタンが無効化されローディング表示になる（アクションはモック）
- 送信失敗時にエラーメッセージが表示される（アクションはモック）
- ダイアログの開閉が正しく動作する

対象外

- 有効な入力で送信できる
  - E2E テストでカバー

## medium サーバーテスト

**対象：** Server Action、Route Handler、サーバー用の関数

E2E では代替しにくいサーバーロジック固有のテストケース。

対象

- UI に表示されない DB 内部状態の検証（論理削除フラグ・タイムスタンプ等）
- 多数のパターンを網羅したいロジック
  - 例：検索が姓・名・かな・電話番号それぞれでヒットすること
  - E2E は代表的なフローのみカバーし、パターン網羅はサーバーテストで担う
- UI を持たないサーバーロジック（バックグラウンドジョブ・バッチ処理等）

対象外

- 登録フォームから保存できる
  - E2E で担保される

## small サーバーテスト

**対象：** Server Action、Route Handler、サーバー用の関数

DB・認証などの外部依存をモック化し、単体で検証できるシナリオ。

対象

- 認証されていない場合に認証エラーが返る
- 一般ユーザーロールの場合に権限エラーが返る
- 必須フィールドが不正なフォーマットのときバリデーションエラーが返る
- ロジックがエラーを投げた場合にサーバーエラーが返る

対象外

- DB に値が正しく保存される
  - medium サーバーテストで対応
